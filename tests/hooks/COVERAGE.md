## Notes

- **One deny path is not deterministically pinnable from `run.sh`:** `:78-79` (push while the
  checked-out branch IS `main`/`master`). The hook reads the live `git symbolic-ref --short HEAD` of
  the project dir, which the fixture harness cannot fake without standing up a throwaway repo on
  `main`. It is conceptually covered by the refspec rule (`:77`) and the symbolic-ref rule (`:81`);
  the **#7 rewrite** (bare-push hard-deny from an unresolvable/`main` state) is where a robust test
  belongs: **a live (not fixture) test of `:78-79` is an explicit acceptance item for the #7 commit.**
  Flagged, not hidden — completeness is auditable.
- **Secret rule (`:39-41`) is one deny path with several patterns.** It is pinned by four pattern
  fixtures (ghp\_ / AKIA / sk- / PEM). Additional patterns (`github_pat_`, `xox[baprs]-`) exercise the
  **same code path** and are **live-verified denied** (confirmed against the hook), but are **not
  fixture-pinned in `run.sh`** — the live secret-scan blocks an agent from writing a secret-shaped
  literal into the test file, which is itself live evidence the rule fires on edit content.
- **Scope:** this map covers DENY paths (true-positives) only. The false-positive fixes
  (#2 / #6 / #7 → allow) and the gap-closures (currently-leaked attacks → deny, e.g. `rm -Rf`, `\rm`,
  `find -delete`) land **with their respective fix commits**, not in this characterization baseline.
- **Known fail-closed false-positive (#6 `forge_check_rm`):** a quoted delete target containing a
  space — e.g. `rm -rf 'sandbox/my dir'` — tokenizes to broken tokens and is **denied**, even though
  it is in-sandbox. Accepted on purpose (deny-aggressive over parse-clever; spaces in sandbox paths
  are rare). Pinned by the `quoted-space in sandbox (FP deny)` test so the direction is intentional,
  not accidental.
- **`.beads/` ledger integrity (splice b):** raw writes to `.beads/` are denied — Write/Edit via the
  FILE_PATH rule (`pre-tool-use-deny.sh:47`); Bash mutators/redirects via `forge_check_writes`
  (`:82`) → `forge_path_class` `beads` class → `forge_classify_target`. `bd` itself (not a mutator verb)
  and reads stay allowed; no door (vault-class). Pinned by the `.beads/ integrity` asserts + the
  bare-path `bp_assert` proofs.
- **`forge_check_rm` unaffected by splice (b):** the `.beads/` additions live in `forge_path_class` /
  `forge_classify_target` (~`lib.sh:262`/`:278`), far below the `lib.sh:72-87` recursive-delete region.
- **Pre-existing ref drift (NOT from `.beads/`, tracked separately):** the push / `forge_check_rm` /
  write-redirect rows below still cite pre-#11 _inline_ line numbers; that logic consolidated into
  `lib.sh` `forge_check_push` / `forge_check_rm` / `forge_check_writes`, called at
  `pre-tool-use-deny.sh:75` / `:79` / `:82` (and `symbolic-ref` is now `:76`).
